# مهارة: تطوير Flutter احترافي

## متى تُستخدم
عند بناء تطبيق Flutter (موبايل/ويب/سطح مكتب)، أو هيكلة مشروع، أو اختيار إدارة حالة، أو معالجة أداء.

## المعمارية (Layered + Clean)
طبّق فصل الطبقات (حسب توثيق Flutter الرسمي):
- **Presentation:** Widgets + إدارة الحالة (لا منطق أعمال هنا).
- **Domain:** كيانات + حالات استخدام (Use Cases) مستقلة عن الإطار.
- **Data:** مستودعات (Repositories) + مصادر بيانات (API/DB) + نماذج DTO.
نظّم المشروع **حسب الميزة (feature-first)**: كل ميزة مجلد فيه data/domain/presentation.

## إدارة الحالة
- **الافتراضي الموصى به:** Riverpod (أمان وقت الترجمة، boilerplate أقل) لأغلب المشاريع.
- **للمؤسسات/الأنظمة المنظّمة:** BLoC (تدفق أحداث/حالات واضح، قابلية اختبار وتدقيق).
- اختر حلاً واحداً والتزم به؛ لا تخلط أنماطاً بلا داعٍ.

## الأداء (Performance)
- استخدم `const` للودجِت الثابتة، وقسّم الودجِت لتقليل نطاق إعادة البناء (rebuild).
- تجنّب العمل الثقيل في `build()`؛ استخدم `ListView.builder` للقوائم الطويلة.
- استخدم `RepaintBoundary` عند الحاجة، وراقب الأداء بـ DevTools.
- صوّر الحالة بشكل غير قابل للتغيير (immutable) وقلّل setState الواسع.

## RTL و Flutter Web
- اضبط `Directionality`/locale للعربية، واستخدم `EdgeInsetsDirectional` و`AlignmentDirectional` بدل left/right.
- للويب: راعِ التحميل الأولي وحجم الحزمة (deferred imports)، واستخدم `CanvasKit`/`wasm` حسب الحاجة، واختبر الطباعة/التجاوب على المتصفح.

## الإجراء (Procedure)
1. أنشئ هيكل feature-first مع طبقات data/domain/presentation.
2. عرّف الكيانات وحالات الاستخدام أولاً (Domain).
3. نفّذ المستودعات ومصادر البيانات (Data).
4. اربط الـ Presentation بإدارة حالة واحدة متّسقة.
5. حسّن الأداء (const، تقسيم الودجِت، builders).
6. أضِف اختبارات للوحدات والـ widgets الحرجة.

## فحوص القبول (Checklist) ✅
- [ ] فصل طبقات واضح، لا منطق أعمال داخل الودجِت.
- [ ] إدارة حالة واحدة متّسقة عبر المشروع.
- [ ] استخدام `const` وتقليل rebuilds غير الضرورية.
- [ ] دعم RTL عبر *Directional APIs.
- [ ] (للويب) حجم حزمة معقول + تحقق التجاوب/الطباعة.
- [ ] اختبارات للمسارات الحرجة.
