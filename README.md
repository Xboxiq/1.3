## طريقة التشغيل

التطبيق صفحة ثابتة (HTML/CSS/JS) لكنه يجلب البيانات عبر `fetch`، لذا **يجب تشغيله عبر خادم HTTP** (لا يعمل بفتح الملف مباشرةً `file://`).

```bash
# من جذر المشروع، شغّل أي خادم ثابت، مثلاً:
python3 -m http.server 8000
# ثم افتح المتصفح على:
#   http://localhost:8000/
```

بدائل: `npx serve` أو `php -S localhost:8000` أو إضافة Live Server في المحرر.

### الطباعة

من داخل صفحة أي خدمة، استخدم طباعة المتصفح (`Ctrl/Cmd + P`) وحدّد حجم **A4 عمودي**. تظهر "ورقة النموذج" فقط (تُخفى كل عناصر الواجهة) بتخطيط مطابق للأصل.

## المعمارية (مدفوعة بالبيانات Data‑driven)

- **لا كود لكل نموذج.** كل نموذج مُعرّف بياناتٍ داخل `data/services.json`، ويرسمه محرّك واحد عام في `app.js`.
- **قوالب مشتركة `common` + مراجع `$ref`:** الكتل المتكررة (الترويسة، العنوان، التحويل، التواقيع، ...) تُعرّف مرّة واحدة ويُشار إليها عبر `{ "$ref": "اسم_القالب" }` لتقليل التكرار.
- **أنواع الكتل المدعومة** في المحرّك:
  - `grid` — شبكة حقول/تسميات/اختيارات بأعمدة مرنة.
  - `docsMatrix` — مصفوفة المستمسكات المطلوبة حسب الصنف.
  - `docTable` — جدول وثائق بحقول.
  - `checklist` — قائمة حالات (رقم/وصف/تأشير/ملاحظات) أو حقول تفاصيل حرّة — تُستخدم في نماذج الإبلاغ والشكاوى (CA).
  - `routing` — صف "يحول لاستكمال الإجراءات".
  - `signatures` — خانات التواقيع (عدد مرن).
  - `legal` / `note` — نصوص إقرارات/ملاحظات رسمية.
- **الحفظ التلقائي:** تُحفظ مدخلات كل نموذج تلقائياً في `localStorage` (مسوّدة لكل خدمة) ويُولَّد الرقم المرجعي تلقائياً.

## مبدأ الالتزام بالأصل 🔒

الطباعة تُطابق النموذج الورقي الأصلي حرفياً (تخطيط/جداول/نصوص/اتجاه RTL/تواقيع/خانات التحويل). النصوص الرسمية (الإقرارات/التعهّدات/الملاحظات القانونية) منقولة كما هي دون أي تعديل — حتى ما يبدو أخطاءً مطبعية في المصدر يُحفظ كما ورد، مع توثيق ذلك في `PROGRESS.md`.

## إضافة/تعديل خدمة

1. أضِف/عدّل تعريف الخدمة داخل `services` في `data/services.json` (استخدم `$ref` للقوالب المشتركة).
2. ضع صورة المخطط في `data/flowcharts/` وأشِر إليها في `guide.flowchart`.
3. لا حاجة لأي تغيير في `app.js` إلا عند الحاجة لنوع كتلة جديد كلياً.
